# Self Notes - Мои решения

## Почему выбрал самописный ReactiveValue вместо UniRx/R3?
Рассматривал R3, но решил написать свой минимальный `ReactiveValue<T>` чтобы:
- Показать понимание внутреннего устройства реактивности
- Избежать лишних зависимостей (R3 весит ~200KB)
- Контролировать момент вызова колбэков (invokeImmediately)

Альтернатива: R3 дает больше операторов (Select, Where, Throttle), но для задачи хватило Subscribe.

## Какие места написал сам без AI:
1. **Логику регенерации** (циклы, sleep при Max) - важно было реализовать эффективно без busy-wait
2. **CompositeDisposable** - своя имплементация чтобы не тащить R3
3. **Подписки и отписки** - ручное управление через `Release()`, без утечек
4. **DI регистрацию** - через VContainer с корректным жизненным циклом

AI использовал для: генерации boilerplate (UIView, интерфейсы), но логику писал сам.

## Почему здесь именно так? (3 ключевых решения)

**1. Почему регенерация через while + Delay, а не через Time.time?**
- `Time.time` требует ручного пересчета в Update() или корутине
- `UniTask.Delay` дает чистое await без MonoBehaviour
- Легко отменить через CancellationToken
- Минус: нет синхронизации с игровым временем (не критично для энергии)

**2. Почему Sleep при MaxEnergy через Delay(1000)?**
- Если поставить Delay(100) - будем проверять 10 раз в секунду впустую
- Если Delay(10000) - слишком долгая реакция на трату энергии
- 1 секунда - золотая середина: быстро реагируем, но не грузим CPU
- Альтернатива: Event на TrySpend + ManualResetEvent, но overkill для этой задачи

**3. Почему UI не через Update(), а через подписки?**
- Update() будет вызываться 60 раз/сек даже когда UI не меняется
- ReactiveValue обновляет UI только при изменении данных
- Это декларативный подход: "когда X изменится - сделай Y"
- Легче отлаживать: знаешь что триггерит обновление
- Подписки автоматически диспозятся через CompositeDisposable
